## 指示

ユーザーと対話しながら要件を整理し、コードベース調査を行い、GitHub issue を作成する。以下の 5 Phase を順番に進める。

引数: `$ARGUMENTS`（省略可。issue の概要を自然言語で記述）

---

## Phase 1: 要件概要のヒアリング

まず以下を確認する。`$ARGUMENTS` で既に提供されている情報はスキップしてよい。

1. **対象リポジトリの確認**
   - `git rev-parse --show-toplevel` でカレントディレクトリが Git リポジトリか確認
   - リポジトリなら「このリポジトリ（`リポジトリ名`）で合っていますか？」と確認
   - リポジトリでなければ対象リポジトリのパスを聞く

2. **変更・機能・修正の概要**
   - 何を実装/変更/修正するのか

3. **動機・背景**
   - なぜこの変更が必要なのか

未提供の項目をまとめて 1 回で質問する（個別に聞かない）。

---

## Phase 2: コードベース調査

Phase 1 で得た情報をもとに、Skill ツールで `/codebase-reader` を呼び出す。

呼び出し時の引数には以下を含める:
- 要件概要（Phase 1 の回答）
- プロジェクトルートのパス
- 特に調べてほしい観点（あれば）

調査結果を受け取ったら、要点をユーザーに共有する:
- 技術スタックの概要
- 関連するファイル・モジュール
- 参考になりそうな既存パターン

---

## Phase 3: 仕様の深掘り

コード調査結果を踏まえ、以下の観点から追加質問を行う。1 回の質問で **1〜3 個** に絞り、段階的に聞く。必要な情報が揃ったら次の Phase に進む。

質問の観点:
- **技術的な設計方針**: 既存パターンを踏襲するか、新しいアプローチを取るか
- **スコープの明確化**: 含むもの/含まないもの、エッジケースの扱い
- **受け入れ基準**: 完了条件、テストのレベル（ユニットテスト、E2E等）
- **優先度・制約**: 後方互換性、フェーズ分けの必要性

全ての観点を必ず聞く必要はない。要件の複雑さに応じて判断する。

---

## Phase 4: issue 下書き

収集した情報を `template.md` のフォーマットに沿って issue の下書きを作成する。

### ルール

- `template.md` の全セクションを埋める必要はない。該当する情報がないセクションは省略する（「なし」「N/A」等は書かない）
- 下書きをユーザーに提示し、レビューを依頼する
- ユーザーの修正要望があれば反映し、OK が出るまで繰り返す

---

## Phase 5: issue 作成

ユーザーの承認を得たら、GitHub に issue を作成する。

### 手順

1. `gh auth status` で GitHub CLI の認証状態を確認
   - 未認証の場合は `gh auth login` を案内して中断

2. `gh label list --limit 100` で既存ラベルを取得し、issue に適したラベルを提案する
   - ユーザーにラベルを選択/変更してもらう
   - ラベルなしでも可

3. issue を作成する
   ```
   gh issue create --title "タイトル" --body "本文" [--label "label1,label2"]
   ```
   - タイトルは簡潔に（70文字以内目安）
   - 本文は Phase 4 の下書き内容

4. 作成された issue の URL を表示する
