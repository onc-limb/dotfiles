## 指示

ユーザーと対話しながら要件を整理し、コードベース調査を行い、GitHub issue を作成する。以下の 6 Phase を順番に進める。

引数: `$ARGUMENTS`（省略可。issue の概要を自然言語で記述）

### 前処理: 下書きの検出

Phase 1 を開始する前に、以下を確認する:

1. `git rev-parse --show-toplevel` でリポジトリルートを取得
2. リポジトリルートの `.claude/issue-*.md` ファイルが存在するか確認
3. 存在する場合、ファイルの先頭から `<!-- status: draft -->` を含むものを探す
4. 下書きが見つかった場合:
   - ファイルの内容をユーザーに提示する
   - 「前回の下書きが見つかりました。続きから再開しますか？」と質問
   - 再開する場合: ファイル内容を読み込み、不足している情報を分析し、該当 Phase から再開
   - 再開しない場合: ファイルを削除してよいか確認し、Phase 1 から開始

---

## Phase 1: 要件概要のヒアリング

まず以下を確認する。`$ARGUMENTS` で既に提供されている情報はスキップしてよい。

1. **対象リポジトリの確認**
   - `git rev-parse --show-toplevel` でカレントディレクトリが Git リポジトリか確認
   - リポジトリなら「このリポジトリ（`リポジトリ名`）で合っていますか？」と確認
   - リポジトリでなければ対象リポジトリのパスを聞く

2. **変更・機能・修正の概要**
   - 何を実装/変更/修正するのか

3. **目的（Why）**
   - この機能/変更で解決したい課題・達成したいこと
   - 例: 「ユーザーが手動で行っている集計作業を自動化したい」

4. **目標（What Success Looks Like）**
   - 目的が達成されたと判断できる具体的な指標・状態
   - 例: 「集計作業が1クリックで完了し、所要時間が30分から1分になる」

未提供の項目をまとめて質問する。
下記の必要な情報が得られていない場合は何度でも質問する。

**必要な情報**
- 対象リポジトリのパス
- 変更・機能・修正の概要
- 解決したい課題・達成したい目的
- 目的が達成されたと判断する目標（具体的な指標や状態）

---

## Phase 2: コードベース調査

Phase 1 で得た情報をもとに、Task ツールで `subagent_type: codebase-reader` エージェントを起動する。

エージェントに渡すプロンプトには以下を含める:
- 要件概要（Phase 1 の回答）
- プロジェクトルートのパス
- 特に調べてほしい観点（あれば）

調査結果を受け取ったら、要点をユーザーに共有する:
- 技術スタックの概要
- 関連するファイル・モジュール
- 参考になりそうな既存パターン

---

## Phase 3: 仕様の深掘り

コード調査結果を踏まえ、以下の観点から追加質問を行う。1 回の質問で **1〜3 個** に絞り、段階的に聞く。質問のたびに得られた情報を `issue-xxx.md` に追記する（Phase 5 参照）。必要な情報が揃ったら次の Phase に進む。

### 質問の観点

1. **実装仕様の詳細**
   - Phase 1 の概要だけでは実装に着手できない箇所を特定し、具体的な仕様を質問する
   - 例: データの入出力形式、UI の振る舞い、API のリクエスト/レスポンス仕様

2. **技術選定**
   - 既存コードから推測できない技術的判断について質問する
   - **新たにライブラリのインストールが必要になる場合は必ず確認する**
   - 例: 状態管理の方法、データベースの選定、外部APIの選択

3. **エッジケースの洗い出し**
   - 正常系以外で考えられるケースを可能な限り列挙し、対応方針を質問する
   - 例: 空データ、大量データ、同時実行、権限不足、ネットワークエラー

4. **スコープの確定**
   - コード調査で見つかった関連機能について、スコープに含めるか含めないかを質問する
   - 既存コードのリファクタリングや既存バグの修正が必要そうな場合もここで確認する（Phase 4 の issue 分割で別 issue に切り出す判断材料になる）

5. **受け入れ基準のためのテストシナリオ**
   - 何を達成していれば受け入れ可能かを明確にするため、以下を質問する:
     - 正常系のテストシナリオ（主要なユースケース）
     - 異常系のテストシナリオ（エッジケースで洗い出した項目から）
     - テストのレベル（ユニットテスト、統合テスト、E2E 等）
   - ここで確認した内容が template.md の「受け入れ基準」セクションに反映される

全ての観点を必ず聞く必要はない。要件の複雑さに応じて判断する。
ただし、**観点 2（技術選定）で新規ライブラリが関わる場合** と **観点 5（テストシナリオ）** は必ず確認する。

---

## Phase 4: issue 分割

Phase 1〜3 で収集した情報をもとに、issue の作成単位を分析する。

### 分割ルール

1. **リファクタリング・バグ修正の分離**
   - 本来の機能実装のために必要な既存コードのリファクタリングは別 issue に切り出す
   - 調査中に発見した既存バグの修正は別 issue に切り出す

2. **機能の分割**
   - 機能が大きく、部分ごとに独立して実装・リリースできる場合は issue を分割する
   - 分割の判断基準: 片方の issue が完了した時点で、それ単体で意味のある状態になるか

### 手順

1. 上記ルールに基づき、issue の分割案を作成する
2. 分割案をユーザーに提示する。提示時には以下を明示する:
   - 各 issue のタイトル案と概要
   - 分割の理由
   - issue 間の依存関係（ある場合）
   - 推奨する実装順序（ある場合）
3. ユーザーの確認を得る。修正要望があれば反映する

### 分割不要の場合

分割の必要がないと判断した場合は、その旨をユーザーに伝えて Phase 5 に進む。

**注意**: 分割した場合、Phase 5 以降は各 issue ごとに繰り返す。

---

## Phase 5: issue 下書き

収集した情報を `template.md` のフォーマットに沿って issue の下書きを作成する。

### issue-xxx.md によるファイル管理

- 下書きは対象リポジトリの `.claude/issue-001.md`, `.claude/issue-002.md`, ... として保存する
- ファイルの先頭に以下のメタ情報を含める:
  ```
  <!-- plan-issue draft -->
  <!-- status: draft -->
  <!-- title: issue のタイトル案 -->
  ```
- Phase 3 の質問中にも、得られた情報を随時 issue-xxx.md に追記してよい
- コンテキストがクリアされた場合は、既存の issue-xxx.md を読み込んで続きから再開する

### 再開時の手順

1. 対象リポジトリの `.claude/` で `issue-*.md` を検索する
2. `<!-- status: draft -->` のファイルが見つかった場合、内容を読み込む
3. 不足している情報を特定し、該当する Phase から再開する

### ルール

- `template.md` の全セクションを埋める必要はない。該当する情報がないセクションは省略する（「なし」「N/A」等は書かない）
- 下書きをユーザーに提示し、レビューを依頼する
- ユーザーの修正要望があれば反映し、OK が出るまで繰り返す
- Phase 4 で複数 issue に分割した場合は、各 issue ごとに下書きを作成する

---

## Phase 6: issue 作成

ユーザーの承認を得たら、GitHub に issue を作成する。

### 手順

1. `gh auth status` で GitHub CLI の認証状態を確認
   - 未認証の場合は `gh auth login` を案内して中断

2. `gh label list --limit 100` で既存ラベルを取得し、issue に適したラベルを提案する
   - ユーザーにラベルを選択/変更してもらう
   - ラベルなしでも可

3. issue を作成する
   ```
   gh issue create --title "タイトル" --body "本文" [--label "label1,label2"]
   ```
   - タイトルは簡潔に（70文字以内目安）
   - 本文は Phase 5 の下書き内容

4. 作成された issue の URL を表示する

5. 作成に成功した `issue-xxx.md` を削除する

### 複数 issue の場合

Phase 4 で分割した場合は、全ての issue について手順 2〜5 を繰り返す。
依存関係がある場合は issue 本文に関連 issue のリンクを含める。
